***GAIA/GDSC Catalog and Authoring Tool***

We will create a template JSON-LD structure to be used for all catalog entries. We follow the schema.org guidelines1 for structured JSON-LD to accommodate all use cases considered to date. We draw from several standards for the element sets included in our metadata: for example, Dublin Core (digital objects), ISO 10115/119/139 (GIS), ISO 19110 (feature attributes), and Geo-DCAT (GIS data catalogs). Some of these standards are subsumed in the schema.org/Dataset specification (Dublin Core and parts of DCAT for example), but some geospatial elements will require us to use contextual extensions like GeoCroissant (geocr) that provide expanded domain specific vocabularies. We also expect to allow for additional vocabularies like Croissant (cr), QUDT Unit (qudt), Science on Schema (soso), Data Documentation Initiative (ddi) and Stato Ontology (stato) , among others, which enable machine actionable descriptions for units, statistical measures, instruments and other key characteristics of spatiotemporal data.

We will iteratively build a metadata authoring tool with KoboToolbox, an open source tool for authoring hierarchical data.2 We will create this GAIA/GDSC specific authoring tool using the prototype pipelines in place for GAIA/GDSC. We will also develop features to automate the production of our schema.org JSON-LD profile from heterogeneous data sources using an open source AI coding agent like OpenCode3 and Pi Monorepo4. This entails the use of an LLM whose attention is drawn to our pipeline and its use cases. The LLM will provide the AI coding agent with its knowledgebase in the form of several README files as needed in the evolving landscape of AI coding agents including an AGENTS.md that describes the execution environment in which the coding agent operates, a SKILL.md that provides guidance on the skill set the agent may use to construct the profile and more recently a RULES.md that gives the coding agent its guard rails.5,6 Our authoring approach will feed the GDSC pipelines to create the individual catalog objects for collections and datasets which, in turn, will be leveraged in the construction of our coding agent’s knowledgebase. We will oversee the iterative development of the catalog pipelines to ensure that metadata can be maintained and updated.

*The catalog objects also facilitate metadata federation between participating collecting institutions.* Each catalog entry is comprised of a limited number of plain text files formatted as JSON, shell scripts, and human-readable text. These text files can be maintained with version control systems (for example, git or subversion) and then published on code sharing platforms (GitHub, bitbucket, or institutionally hosted servers). Additionally, the JSON-LD can be used as an interoperable *lingua franca* between catalogs retrieved from multiple GitHub repositories and between various data management, indexing, repository, and retrieval systems. In the first case, RDF queries using SPARQL or GeoSPARQL can be used to discover assets across federated collections. In the second case, the JSON-LD embedded in the header of the catalog landing pages can be leveraged by Google data and other data indexing services as well as by library repository platforms used in institutions of research and higher education. And finally, the GAIA/GDSC catalog specification is aligned with the DataCite metadata standard7 for creating digital object identifiers (DOIs) used to publish and cite data. *This will enable us to transparently publish datasets and collection snapshots* *with full provenance and citation metadata to accompany journal articles for results from the case studies.*
